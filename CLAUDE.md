## What This Repository Is

A universal, self-propagating agent framework for autonomous AEM (Adobe Experience Manager) feature development. It consists of agent definitions, a master pipeline, code pattern templates, and tooling — designed to be copied into any AEM project via `install.sh`. It is **not** an AEM project itself; it is the distribution source for the `.agent/` folder.

## Key Commands

```bash
# Install into a target AEM project
bash install.sh /path/to/target-aem-project

# Generate project.yaml for a target project (run from within that project)
bash setup.sh

# Normalize a Jira export into a dev plan workbook + JSON
python3 jira_plan.py <input.csv|xlsx> [--priority "High,Critical"] [--issue-type "Story"] [--include-missing-ac]

# jira_plan.py optional dependency (for .xlsx output)
pip install openpyxl
```

There are no build/test/lint commands for this repo itself. The Maven commands in AGENT.md and REPO_CONTEXT.md (`mvn clean install -T1`, etc.) are templates meant to run inside target AEM projects, not here.

## Architecture

### Agent Hierarchy (top-down orchestration)

```
jira-planner ─── parses Jira export, generates dev plans, delegates per story
    └── aem-fullstack ─── orchestrates Plan → Build → QA → PR per story
            ├── aem-feature ─── implements code (8-step pipeline from AGENT.md)
            └── aem-qa ─── validates rendering, manages fix-retest loops
```

- **aem-feature.agent.md**: Core builder. Two modes: Feature Mode (full 8-step pipeline) and QA Fix Mode (fix specific defects).
- **aem-qa.agent.md**: QA specialist. Two modes: Report-Only and Autonomous-Fix (delegates fixes to aem-feature, retests up to 10 cycles).
- **aem-fullstack.agent.md**: End-to-end orchestrator. 5 phases: Plan → Build → QA → PR → Report. Hard stop after 5 QA fix cycles. One PR per story, sequential processing with branch isolation.
- **jira-planner.agent.md**: Planner. Parses CSV/XLSX, normalizes via `jira_plan.py`, generates dev plans. Has Full Mode (deep repo search) and Light Mode (skip file discovery, let aem-feature handle it).

### Core Files

| File | Role |
|------|------|
| `AGENT.md` | Master 8-step pipeline (load config → understand → plan → implement → build → test page → test → autofix → report) |
| `REPO_CONTEXT.md` | Universal AEM code patterns with `{{placeholder}}` syntax (components, dialogs, HTL, Sling Models, OSGi, servlets, schedulers, clientlibs, tests) |
| `QUICKSTART.md` | User-facing setup and usage guide |
| `jira_plan.py` | Standalone CLI tool: Jira CSV/XLSX → normalized workbook + JSON |
| `install.sh` | One-command installer into target AEM projects |
| `setup.sh` | Project scanner that auto-generates `.agent/project.yaml` |

### Placeholder System

All `{{...}}` placeholders in AGENT.md and REPO_CONTEXT.md resolve from `.agent/project.yaml` (generated by `setup.sh` in the target project). Examples: `{{project.groupId}}`, `{{java.basePackage}}`, `{{jcr.contentRoot}}`, `{{modules.core}}`, `{{build.deployProfile}}`.

### AEM Version Abstraction

The framework auto-detects and adapts to AEM type (`cloud`, `lts`, `ams`, `on-prem`), adjusting Java version, OSGi config format (XML vs JSON), deploy profiles, servlet annotations, and replication APIs.

## Working with jira_plan.py

The only Python file. Uses stdlib + optional `openpyxl`. Key entry point is `main()` → `normalize_export()` → `build_workbook()` / `write_json()`.

Notable design choices:
- Smart header detection: scans rows to find the one containing Jira field names (handles exports with metadata rows above)
- AC extraction fallback: if no Acceptance Criteria column, regex-extracts from Description field (looks for "Acceptance Criteria:", "AC:", or Gherkin patterns)
- Generates two prompt styles per ticket: a detailed `aem-feature` prompt and a shorter `copilot_prompt`
- Deduplicates by Jira Key

## Critical Rules Embedded in the Framework

- **Always `-T1`** for Maven builds (single-threaded prevents ConcurrentModificationException)
- **Always `sync`** before Maven builds (flush filesystem buffers)
- **Step 5a is mandatory**: agents must inspect 2-3 existing content pages before creating test pages
- **Never modify existing pages** — always create dedicated test pages
- **One PR per story** — never bundle multiple stories
- **Sequential story processing** — ensures branch isolation
